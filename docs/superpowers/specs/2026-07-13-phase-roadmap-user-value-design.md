# Phaseロードマップ利用価値追記 設計メモ

## 目的

`docs/code-knowledge-tool-requirements-v2.md`のPhaseロードマップを読んだとき、機能名だけでなく「そのPhaseが完了すると何が便利になるか」を利用者目線で理解できるようにする。

Phase 0とPhase 1は基盤構築・最小実用版として既に完了しているため、新しい説明はPhase 2〜4だけに追加する。

## 記述形式

14章のPhase 2〜4それぞれの見出し直下へ、次の順序で追記する。

1. `このPhaseで便利になること` — 利用者が得る価値を短い段落で説明する。
2. `利用イメージ` — Agentの具体的な利用場面を1例示す。
3. 既存の機能一覧 — 実現手段として現在の箇条書きを残す。

## Phase別の内容

### Phase 2: 鮮度検証

- 保存済みナレッジが現在のコードでも信用できるか判定できる。
- 全体を毎回調べ直さず、変更された根拠だけ再調査できる。
- 未コミット変更がある場合は、保存済み判定を過信せず該当ファイルを直接確認できる。
- 利用例は、過去ナレッジを検索したAgentが`validate_knowledge`を実行し、判定に応じて再調査範囲を変える流れとする。

### Phase 3: 差分保存

- 過去の調査時点から何が変わったかを比較し、変更内容を再利用可能な履歴として残せる。
- ナレッジ更新の経緯を後の調査で参照できる。
- 不要な旧バージョンやProject ID変更に伴うナレッジの管理も行える。
- 利用例は、仕様変更後にAgentが差分を調査し、ユーザー同意後に比較結果を保存する流れとする。

### Phase 4: 検索強化

- 表記や言い回しが異なる質問でも、意味的に近いナレッジを見つけやすくなる。
- C#コードでは、整形など意味に影響しない変更による鮮度判定の誤検知を減らせる。
- Phase 4は実運用で検索精度または鮮度判定精度の不足が確認された場合のみ実施する。
- 利用例は、保存時と異なる用語で質問しても関連ナレッジを検索できる場面とする。

## スコープ外

- Phase 0・1への価値説明の追加
- Phase 2〜4の機能範囲、順序、受け入れ条件の変更
- Phase 4を必須Phaseへ変更すること
- 新しいToolやデータモデルの追加

## 完了条件

- Phase 2〜4の各節だけに利用価値と利用例が追加されている。
- 説明が既存の詳細要件と矛盾しない。
- 機能一覧は削除せず、価値を実現する手段として残っている。
- Phase 2とPhase 3の境界、およびPhase 4が任意であることが読み取れる。
